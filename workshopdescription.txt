[h1] 过月性能优化 [/h1]

当前版本重点是为原后端建立只存在于内存中的高速缓存结构：在不延迟世界结算、不跳过原版逻辑、不写入自定义存档数据的前提下，bypass 原版过月中的低效扫描路径，并重写为基于本 Mod 高速缓存表的高效反向查询 / 候选上界查询。

这些缓存不写盘，退出世界或切档后会释放；诊断样本中主要缓存命中率普遍保持在 90% 以上，部分目标索引、matcher 和静态 planning graph 缓存命中率可突破 99%。

对密闻数量多、NPC 行动规划重、存档体积大的后期档，当前测试样本中过月计算耗时通常可压缩到原版的 50%-15% 左右，也就是约 2-8 倍的过月计算加速；具体收益取决于存档热点分布、硬盘性能和其它 Mod。

本 Mod 不接管、延迟或新增原版保存。

[h3][b]不写入自定义存档数据，不改变原版世界结算屏障；可在已有存档中自由加入/取消本mod。[/b][/h3]

[h2]优化更新#260628[/h2]
这一版继续整理 NPC 月行动规划优化，把主/副目标规划之间的缓存屏障收束为更清晰的两段式结构：

primary planning 读取冻结快照，热路径只读，尽量避免 worker 线程锁竞争。

primary ApplyAll 串行执行后记录 location / relation / inventory delta，不在 planning 热路径里抢锁更新大表。

primary ApplyAll 完成后，根据 delta 局部发布更新后的目标索引 / 关系预过滤 / 物品持有人预过滤 / matcher 依赖版本，供 secondary planning 使用；只有覆盖面过大或状态不可证明安全时才回退到更保守的路径。

这一版还补充了静态 planning graph 邻接缓存，用于减少完整新规划时的条件/效果节点扫描；对后续复用已有规划的月份，热点仍主要在 PrepareContext、FilterActionTargets 和 matcher 判断上。

诊断日志新增或细化了 delta publish、affected block / area、target lookup、target matcher、relation / item prefilter、planning graph cache 等信息，方便判断当前存档到底卡在扫描、匹配、执行、组织更新还是保存阶段。

本次更新的重点不是改变 NPC 行为，而是把高频查找路径整理成“规划阶段冻结、写回阶段记录、下一阶段局部更新”的结构。对于 NPC 行动规划占比较高的后期档，预计可以进一步降低主/副目标规划阶段的等待时间，并让之后的维护更稳。
